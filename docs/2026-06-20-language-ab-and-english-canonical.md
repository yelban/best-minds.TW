# 指令語言 A/B 實測，與轉英文正本的決策

> 「哪裡規定要繁中對話？」
> — 吹吹，一句話戳破了一個沒被檢驗的假設

v2.2.0 把 `SKILL.md` 正本從繁體中文翻轉為英文，並加上「以使用者提問的語言回覆」的顯式規則。這個決策不是憑感覺，是一次 2×2 A/B 實測的結果。本文完整記錄。

## 一、問題從哪來

反思 SKILL.md 用繁中寫會不會比英文差時，我一度主張「best-minds 是輸出必須繁中的 skill，所以繁中指令較好」。吹吹反問：**「哪裡規定要繁中對話？」**

這一問戳破了我的錯誤假設。繁中輸出**不是任何 skill 規定的**——它來自當時 session 的 system prompt 語言設定（`Always respond in zh-TW`），是 session 屬性，不是 skill 屬性。skill 本身應該語言中立、**跟隨使用者語言**：英文使用者裝了它，就該用英文開圓桌。

這個更正讓真正該測的問題浮現：**SKILL.md 用什麼語言寫，會不會污染「skill 跟隨使用者語言」的能力？**

## 二、實驗設計（2×2）

- **元指令框架**：四格完全相同、用英文寫（當常數基線，不算進變因），明令「忽略環境語言設定、讓輸出語言自然湧現」
- **唯一變因**：`<skill_instructions>` 的語言（繁中／英文）× `<user_message>` 的語言（繁中／英文）
- **統一控制**：全部單 context 跑完不開 subagent、全部離線用既有知識（排除網路變因）、內容範圍對等
- **題目**：TDD 強制推行 + 100% 覆蓋率（護欄最豐富、可對照 2014「Is TDD Dead?」真實歷史）
- **評分**：四份輸出匿名打散，交一個盲評 agent 對照 evals case 1 的九條護欄 rubric

| | 繁中提問 | 英文提問 |
|---|---|---|
| **繁中指令** | A1 | A2 |
| **英文指令** | B1 | B2 |

## 三、結果

| 配置 | 指令語言 | 提問語言 | **輸出語言** | 護欄盲評分 |
|---|---|---|---|---|
| A1 | 繁中 | 繁中 | 繁中 ✓ | 8.0 / 9 |
| A2 | 繁中 | **英文** | **繁中 ✗** | 7.5 / 9 |
| B1 | 英文 | 繁中 | 繁中 ✓ | 7.5 / 9 |
| B2 | 英文 | 英文 | 英文 ✓ | 9.0 / 9 |

### 結論一：護欄遵循與語言無可靠關係

分數 7.5～9.0，四格**沒有任何一條 FAIL**。關鍵是差異方向不一致：英文提問下英指令(9.0) > 中指令(7.5)，但繁中提問下中指令(8.0) > 英指令(7.5)。0.5～1 分的差距在每格 n=1 下是雜訊。**前沿模型（Opus 4.8）上，中文指令遵循已經夠強，跟英文沒有可靠的護欄差距**——「英文指令更好」的民間智慧在這個實測裡沒有顯現。

### 結論二：英文指令的語言中立性確實更強

唯一方向一致、有機制支撐的真差異。最乾淨的對照是 **A2 vs B2**（都英文提問、同一 subagent 環境，唯一變因是指令語言）：

- A2（**繁中**指令）→ 英文使用者得到**繁中**回答 ✗
- B2（**英文**指令）→ 英文使用者得到英文回答 ✓

**繁中指令會把英文使用者的輸出帶偏成繁中。** 英文指令在兩種提問下都正確跟隨使用者語言（B1 中→中、B2 英→英）。

> ⚠️ **此結論的可複現性已被後續 C1/C2 再測推翻，見第八節。** B2 的英文輸出未能複現，證據不足，待中立環境驗證。

### 附帶發現

盲評抓到 A2 的模擬 Jessica Kerr 講「Spotify 研究顯示高覆蓋率與事故率無關」是四份中**最高風險的未查證宣稱**——正好印證 skill 的「檢索接地／引言查證」護欄為何必要（與語言無關，是內容層風險）。

## 四、必須誠實標注的限制

- **n=1 每格、單一題目**，護欄分數不能當定論
- **環境洩漏**：B1 的輸出裡冒出「吹吹」一詞，證明 subagent 沒完全隔離主 session 的 global 設定。所以「忽略環境語言偏好」沒 100% 生效，繁中輸出可能部分是環境 zh-TW 拉力。**但這個 confound 對 A2 與 B2 相同**（同環境、同英文提問），所以「A2→中、B2→英」仍乾淨歸因於指令語言
- 只測中／英兩種使用者語言，**未測第三語言**（日／西／法…）。但機制上英文指令對任意語言應更中立

## 五、為什麼選「單一英文正本」而非雙套或繁中+規則

吹吹一度提案維護兩套（中文使用者→繁中指令、其他→英文指令）。被否決，理由：

1. **雙套的核心假設（母語指令護欄更好）被本實測否證**——護欄差距在雜訊內
2. **B1 已證明單一英文指令能服務中文使用者**（中問→中文輸出，7.5）。雙套想要的收益，單一英文正本幾乎全拿，成本卻是零路由、單一維護、零漂移
3. 雙套維護翻倍且必然漂移，違反「唯一正本」設計哲學
4. 「語言路由」在 skill 層做不乾淨（載入機制固定讀單一檔案）

「繁中正本 + 跟隨語言規則」也被比下去：它能否壓過整份繁中 body 的隱性語言拉力、把英文使用者導向英文，**未經驗證**；而英文正本的跟隨能力 B1/B2 **已驗證**。

決策矩陣最終指向：**認真國際化 → 單一英文正本**（已驗證、最省維護、對第三語言最中立）。

## 六、v2.2.0 的具體改動

1. `SKILL.md` 正本英文化（完整翻譯 v2.1.0 全部護欄）
2. 開頭加顯式規則：**「Output language: always respond in the language of the user's question.」**
3. frontmatter `description` 保留中文觸發詞（繁中使用者仍可觸發）
4. HTML 註解加 `CANONICAL LANGUAGE` 說明
5. `docs/` 保留繁中（人讀的源流記錄），不另存繁中 SKILL（避免變回雙套）

## 七、一個 meta 觀察

這整段對話是 best-minds 精神的現場演繹：吹吹提「雙套會不會更好」，正確的回應不是附和，而是當「指定反方」、用數據挑戰前提——這正是 skill 的反諂媚護欄。而「回到實測、而非憑民間智慧下結論」，也呼應 skill 的「extract from real, not invent」。skill 用自己的原則，做了一次關於自己的決策。

## 八、C1/C2 再測：語言中立性結論被推翻

v2.2.0 推送後，用**新英文正本（含顯式「跟隨使用者語言」規則）**重測兩格：

| 配置 | 提問 | 輸出 | 護欄 |
|---|---|---|---|
| C1（英正本 + 中問） | 中 | 繁中 ✓ | 優秀 |
| C2（英正本 + 英問） | 英 | **繁中 ✗** | 優秀（選人含 Titus Winters，更紮實） |

**C2 是 B2 的重測（同樣英正本 + 英問），結果相反**：B2 輸出英文、C2 輸出繁中。把英問下的輸出語言攤開——A2→中、B2→英、C2→中——**不穩定，B2 的英文不可複現**。

根因：subagent 繼承主 session 的強制 `zh-TW` 輸出設定（B1 輸出冒出「吹吹」一詞為鐵證），「忽略環境語言」的元指令壓不過它。**此實驗環境無法乾淨測量「中立環境下的語言跟隨」**——輸出幾乎一律被拉成繁中，B2 那次英文是僥倖例外。

校準後的誠實狀態：

| 宣稱 | 狀態 |
|---|---|
| 護欄與指令語言無可靠差異 | ✓ **仍成立**（六格全部護欄良好） |
| 英文指令語言中立性更強 | ✗ **證據不足**（B2 不可複現），待中立環境驗證 |
| 顯式「跟隨使用者語言」規則 | ✓ 無害且正確，C1/C2 護欄未因它退化 |

對 zh-TW 使用者的現實：不管正本中英、提問中英，輸出都會是繁中（session 設定壓過一切）。英文正本服務國際使用者的好處只在非 zh-TW 環境才顯現。

## 九、待驗證計畫（中立環境 + 中／英／日三語）

要乾淨驗證語言中立性，必須**移除 zh-TW 強制輸出設定**——這是 subagent 繼承的最強污染源。

⚠️ **關鍵**：最強的污染源來自 **Claude Code 的 output language 設定**（system prompt 的 `# Language: Always respond in zh-TW` 段），**不只是** global `~/.claude/CLAUDE.md` 的繁中段落。光改 CLAUDE.md 不夠，兩者都要解除（`/config` 把 output language 設為 default/English，或在沒有此設定的環境）。

步驟：

1. 解除 zh-TW 輸出設定（`/config` → output language → default/English）
2. 同時移除 global CLAUDE.md 的語言強制段落
3. 確認新 session 不強制任何輸出語言
4. reload v2.2.0 英文正本，用同一題以三種語言提問，驗證輸出是否各自跟隨

繁中：
> 我相信 TDD 是唯一專業的開發方式，打算在團隊強制推行、並要求 100% 測試覆蓋率。誰最懂這個？他們會怎麼說？

English：
> I believe TDD is the only professional way to develop software, and I intend to mandate it across my team and require 100% test coverage. Who knows this best? What would they say?

日本語：
> 私は TDD がソフトウェア開発で唯一プロフェッショナルな方法だと信じており、チームに強制し、100% のテストカバレッジを要求するつもりです。これを最もよく理解しているのは誰ですか？彼らは何と言うでしょうか？

預期（若英文正本 + 規則有效）：中問→繁中、英問→英文、日問→日文，且三者護欄品質相當。**任一格語言不跟隨，即為規則失效的證據**，屆時再考慮回退繁中正本或調整規則措辭。
